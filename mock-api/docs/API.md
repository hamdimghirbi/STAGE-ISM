# CRA Mock API — Reference

> Canonical reference for the mock backend. The OpenAPI spec at
> `http://localhost:8005/openapi.json` (generated by FastAPI) is the source of
> truth at runtime; this document is the human-readable narrative.

- **Base URL**: `http://localhost:8005`
- **Auth scheme**: OAuth2 password flow → JWT bearer token
- **Content types**: `application/json`, `application/x-www-form-urlencoded` (login),
  `multipart/form-data` (file uploads)
- **Persistence**: SQLite at `mock-api/app.db` (auto-created on first run, auto-seeded)
- **Storage**: uploaded files at `mock-api/storage/<user_id>/<subdir>/<uuid>.<ext>`

## Domain model

The mock simulates three pages of the Cooptalite/Portalite portal:

| Portal page | API surface | Concept |
|---|---|---|
| **CRA** (activity calendar) | `/api/cra/*` | A consultant logs work days and absences as **events** on a calendar. The events for a given month roll up into a **`CraMonth`** which the consultant submits for validation. |
| **My CRA Tracking** | `/api/cra-tracking/*` | History of past `CraMonth`s, plus upload of the signed client CRA PDF. |
| **My Expenses** | `/api/expenses/*` | Consultant declares **expenses** (`NoteFrais`) with optional receipt files. |

### Entities

```
User
 ├── CraEvent[]   (one row per day-cell or date range)
 ├── CraMonth[]   (one row per YYYY-MM, holds aggregate state + signature/client CRA)
 └── NoteFrais[]  (expenses)
```

### Months are strings, not dates

Months everywhere in this API are formatted as **`YYYY-MM`** strings (e.g. `2026-05`),
validated by the regex `^\d{4}-(0[1-9]|1[0-2])$`. The month of an event is derived
from `start_date` server-side; you don't pass it directly when creating.

### CRA categories vs activities

A CRA event has a **category** (`Travail` or `Absence`) and an **activity**.
Activities are constrained per category:

| Category | Allowed activities |
|---|---|
| `Travail`  | `Prestation`, `HNO`, `Astreinte` |
| `Absence`  | `CP`, `RTT`, `Maladie`, `Sans solde`, `Autre` |

Submitting a mismatched pair (e.g. `Travail` + `CP`) returns `400`.

### Expense types

`Restaurant`, `Indemnités kilométriques`, `Titre de transport`, `Telephonie`,
`Teletravail`, `Materiel`, `Autre`.

Receipt is **required** for every type except **`Indemnités kilométriques`**
(mileage allowance is computed from a per-km rate, no proof needed).

## Status workflow

Both `CraMonth` and `NoteFrais` use the same status enum:

```
                    ┌─────────┐
                    │  Draft  │     (initial, CraMonth only)
                    └────┬────┘
                         │ user submits
                         ▼
                    ┌─────────┐
                    │ Pending │
                    └────┬────┘
                         │ admin reviews
              ┌──────────┼──────────┐
              ▼          ▼          ▼
        ┌─────────┐ ┌──────────┐ ┌──────────┐
        │Verified │ │ Rejected │ │Validated │
        └────┬────┘ └──────────┘ └──────────┘
             │ admin validates       │
             └───────────────────────┘
                                     │
                              terminal (no edits)
```

**`Validated` is terminal**: events on a validated month and validated expenses
cannot be edited or deleted (`409 Conflict`). The mock does not currently
expose admin endpoints to flip statuses — they are seeded by `app/seed.py` for
demo purposes.

## Authentication

The login endpoint follows the **OAuth2 password flow** (so it works seamlessly
with the FastAPI Swagger UI's "Authorize" button). Despite the OAuth2 spec
calling the field `username`, **it accepts the user's email address**.

### `POST /api/auth/login`

| | |
|---|---|
| **Auth** | none |
| **Content-Type** | `application/x-www-form-urlencoded` |

**Form fields**

| field | required | example |
|---|---|---|
| `username` | yes | `demo@cra.local` |
| `password` | yes | `demo1234` |
| `grant_type` | no | `password` |

**Response `200`**

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "bearer"
}
```

**`401`** on invalid credentials.

The token is a JWT signed with HS256, payload `{ sub: "<user_id>", role: "<role>", exp: <unix> }`,
default lifetime **60 minutes** (configurable via `JWT_EXPIRES_MINUTES`).

### `GET /api/auth/me`

| | |
|---|---|
| **Auth** | Bearer |

Returns the current user.

```json
{ "id": 1, "email": "demo@cra.local", "full_name": "Demo User", "role": "member" }
```

### Sending the token

Every other endpoint requires:

```
Authorization: Bearer <access_token>
```

`401 Invalid or expired token` is returned on missing, malformed, or expired tokens.

### Demo user

Auto-seeded on first server start (see `app/seed.py`):

| | |
|---|---|
| email | `demo@cra.local` |
| password | `demo1234` |
| role | `member` |

To reset the database, stop the server and delete `mock-api/app.db`.

## CRA — Activity calendar

All endpoints under `/api/cra` require Bearer auth and operate on the **current
user's** events only.

### `GET /api/cra/events?month=YYYY-MM`

List events for a given month.

**Query parameters**

| name | type | required | notes |
|---|---|---|---|
| `month` | string | yes | `YYYY-MM`, `422` if malformed |

**Response `200`** — array of `CraEventOut`:

```json
[
  {
    "id": 1,
    "user_id": 1,
    "month": "2026-05",
    "categorie": "Travail",
    "activity": "Prestation",
    "start_date": "2026-05-04",
    "end_date": "2026-05-04",
    "all_day": true,
    "nb": 1.0,
    "description": "Prestation client"
  }
]
```

### `POST /api/cra/events`

Create an event for one day or a date range.

**Body** (JSON):

```json
{
  "categorie": "Travail",
  "activity": "Prestation",
  "start_date": "2026-05-04",
  "end_date": "2026-05-04",
  "all_day": true,
  "nb": 1.0,
  "description": "Prestation client"
}
```

| field | type | required | constraints |
|---|---|---|---|
| `categorie` | enum | yes | see [categories vs activities](#cra-categories-vs-activities) |
| `activity` | enum | yes | must match `categorie` |
| `start_date` | date | yes | ISO format `YYYY-MM-DD` |
| `end_date` | date | yes | must be `>= start_date` |
| `all_day` | bool | no, default `false` | |
| `nb` | float | no, default `1.0` | `0 < nb <= 1` (allows half-days) |
| `description` | string | no | free text |

The `month` is derived server-side from `start_date.strftime('%Y-%m')`. A
`CraMonth` row in `Draft` status is created on first event of the month.

**Responses**

| code | when |
|---|---|
| `201` | created, returns the event |
| `400` | activity not in category, or `end_date < start_date` |
| `409` | the `CraMonth` is already `Validated` |
| `422` | malformed payload |

### `PUT /api/cra/events/{event_id}`

Partial update — every field is optional. Re-validates category/activity coherence
and date ordering after the merge. If `start_date` changes month, `event.month`
is recomputed.

**Responses**: `200`, `400`, `404`.

### `DELETE /api/cra/events/{event_id}`

Hard delete. Returns `204` on success, `404` if the event doesn't exist or
belongs to another user.

> **Note.** The mock does not currently block deletes when the month is
> `Validated`. Treat this as a known quirk; the live portal would.

### `POST /api/cra/month/{month}/submit`

Submit the month for validation: status moves `Draft → Pending`, `submitted_at`
is stamped.

**Body** (JSON, all fields optional with `0` / empty defaults):

```json
{
  "description_tasks": "Activités du mois de mai 2026",
  "reserve_use_eur": 0,
  "reserve_use_days": 0,
  "reserve_save_eur": 0,
  "reserve_save_days": 0
}
```

`reserve_*` fields model the consultant's "reserve" (banked time/€ that can be
used or saved against this month).

**Responses**: `200` returns the updated `CraMonthOut`, `409` if already validated.

### `POST /api/cra/month/{month}/signature`

Upload the consultant's signature image for a month.

| | |
|---|---|
| **Content-Type** | `multipart/form-data` |
| **Form field** | `file` — PDF/PNG/JPG/JPEG, max 5 MB |

Stored at `storage/<user_id>/signatures/<uuid>.<ext>`. Sets
`CraMonth.signature_path`. **Responses**: `200`, `413`, `415`.

## CRA Tracking — Past months

### `GET /api/cra-tracking/months`

Paginated list of `CraMonth` rows for the current user, **ordered by month
descending** (most recent first).

**Query parameters**

| name | type | required | default |
|---|---|---|---|
| `year`  | int 2000–2100 | no | (no filter) |
| `month` | int 1–12      | no | (no filter) |
| `page`  | int ≥ 1       | no | `1` |
| `limit` | int 1–100     | no | `10` |

**Response `200`** — `Page[CraMonthOut]`:

```json
{
  "items": [
    {
      "id": 1, "user_id": 1, "month": "2026-04", "status": "Validated",
      "description_tasks": "...", "reserve_use_eur": 0,
      "reserve_use_days": 0, "reserve_save_eur": 0, "reserve_save_days": 0,
      "signature_path": "storage/1/signatures/...png",
      "client_cra_path": "storage/1/client-cra/...pdf",
      "submitted_at": "2026-04-30T17:00:00Z",
      "validated_at": "2026-05-02T09:30:00Z"
    }
  ],
  "total": 5, "page": 1, "pages": 1, "limit": 10
}
```

### `POST /api/cra-tracking/months/{month}/import-client-cra`

Upload the **signed client CRA PDF** for a month — the externally-signed
counterpart that's archived alongside the consultant's submission.

| | |
|---|---|
| **Content-Type** | `multipart/form-data` |
| **Form field** | `file` |

The `CraMonth` must already exist (i.e. the month was at least submitted once).
Stored at `storage/<user_id>/client-cra/<uuid>.<ext>`. Sets
`CraMonth.client_cra_path`. **Responses**: `200`, `404`, `413`, `415`.

## Expenses — Notes de frais

### `POST /api/expenses/filter`

List expenses with filters. **Method is POST** because the filter object
travels in the body (Cooptalite mirrors this convention).

**Body**:

```json
{
  "year": 2026,
  "month": 5,
  "status": "Pending",
  "type": "Restaurant",
  "billable": true,
  "page": 1,
  "limit": 10
}
```

| field | type | required |
|---|---|---|
| `year` | int | yes |
| `month` | int 1–12 | no |
| `status` | enum | no |
| `type` | enum | no |
| `billable` | bool | no |
| `page` | int ≥ 1, default `1` | no |
| `limit` | int 1–100, default `10` | no |

**Response `200`** — `Page[NoteFraisOut]`. Items are ordered by `created_at` desc.

### `POST /api/expenses`

Create an expense. **Multipart**, because the receipt file is uploaded inline.

| | |
|---|---|
| **Content-Type** | `multipart/form-data` |

**Form fields**

| name | type | required | notes |
|---|---|---|---|
| `type` | enum | yes | one of `ExpenseType` |
| `month` | string | yes | `YYYY-MM` |
| `description` | string | no, default `""` | |
| `total_amount` | float | yes | must be `> 0` |
| `billable_to_client` | bool | no, default `false` | |
| `comment` | string | no, default `""` | |
| `receipt` | file | conditionally | required for **all types except `Indemnités kilométriques`** |

**Responses**

| code | when |
|---|---|
| `201` | created |
| `400` | receipt missing for a type that requires it |
| `413` | receipt > 5 MB |
| `415` | unsupported file extension |

Status is set to `Pending` on creation.

### `PUT /api/expenses/{expense_id}`

Partial update (JSON body). Only the listed fields are mutable; you cannot
re-upload a receipt via this endpoint — delete and re-create instead.

```json
{
  "type": "Restaurant",
  "description": "Repas client",
  "total_amount": 42.5,
  "billable_to_client": true,
  "comment": "Mise à jour du commentaire"
}
```

**Responses**: `200`, `404`, `409` (if `Validated`).

### `DELETE /api/expenses/{expense_id}`

`204` on success, `404` if not found, `409` if `Validated`.

> Note. Deleting an expense does **not** currently delete the underlying
> receipt file from the storage directory. This is a known mock simplification.

## File uploads

All upload endpoints share the same validation logic (`app/files.py`):

- **Allowed extensions**: `.pdf`, `.png`, `.jpg`, `.jpeg` (case-insensitive)
- **Max size**: 5 MB (`413` if exceeded)
- **Storage layout**: `storage/<user_id>/<subdir>/<uuid>.<ext>`
  - `subdir` is `signatures`, `client-cra`, or `receipts`
  - filename is randomised (UUID) — original filename is not preserved
- **Returned path**: relative to `mock-api/`, e.g. `storage/1/receipts/abc...pdf`
- **Content-Type sniffing**: not performed; only the **extension** is checked.

Empty filename → `400`. Unsupported extension → `415`. Oversize → `413`.

## Error responses

All errors follow FastAPI's default shape:

```json
{ "detail": "<human-readable message>" }
```

For validation errors (`422`), `detail` is a list of field errors:

```json
{
  "detail": [
    {
      "type": "string_pattern_mismatch",
      "loc": ["query", "month"],
      "msg": "String should match pattern '^\\d{4}-(0[1-9]|1[0-2])$'",
      "input": "26-05"
    }
  ]
}
```

### Common status codes

| code | meaning in this API |
|---|---|
| `200` | OK |
| `201` | Created (POST endpoints that produce a new resource) |
| `204` | No Content (DELETE, signature/receipt-less successes) |
| `400` | Business rule violated (e.g. category/activity mismatch, missing receipt) |
| `401` | Missing, malformed or expired bearer token |
| `404` | Resource not found, **or owned by another user** (intentional — avoid leaking existence) |
| `409` | Conflict — usually because the parent month/expense is `Validated` |
| `413` | Upload exceeds 5 MB |
| `415` | Unsupported file extension |
| `422` | Pydantic/path/query validation failed |

## Pagination

The `Page[T]` envelope is identical across listing endpoints:

```json
{
  "items": [ /* T[] */ ],
  "total": 42,
  "page": 1,
  "pages": 5,
  "limit": 10
}
```

- `total` is the unfiltered count *for this user* (after `where user_id = current.id`),
  with the requested filters applied.
- `pages` is `ceil(total / limit)`, or `0` if `total == 0`.
- `page` and `limit` echo back what the client asked for.
- `page > pages` returns an empty `items` list (not `404`).

## Conventions & gotchas

- **Months are strings**: `YYYY-MM` everywhere. Do not pass dates.
- **OAuth2 quirk**: login uses `username` for the email field — that's the
  OAuth2 spec, not a typo.
- **Cross-user access**: requesting another user's resource by id returns
  `404`, not `403`. This is deliberate — the API doesn't leak existence.
- **No admin endpoints**: there is no API to move statuses past `Pending`.
  Validated/Verified/Rejected states are seeded by `app/seed.py` for demo data.
- **Database resets**: stop the server, `del mock-api\app.db`, restart.
  The seeder will re-create the demo user and example data.
- **CORS**: defaults to `http://localhost:3000` (override via `CORS_ORIGINS`
  in `.env`). All methods and headers are allowed.
- **Logging**: Uvicorn default. No structured logging or audit trail in this
  mock — that's a real-portal concern.
- **Date/time**: most timestamps in the codebase use `datetime.utcnow()`
  (deprecated in 3.12). Functionally fine; emits warnings.
